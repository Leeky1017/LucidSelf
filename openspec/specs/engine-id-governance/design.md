# 引擎ID治理 Design

## 设计意图
本设计文档用于固定「引擎ID治理」在规格层面的设计立场：契约、数据完整性、门禁与审计。禁止写实现细节。

## 依赖关系
- 上游依赖：无

## 必备标识符（最小集）
- `engine_id`, `version_id`

## 关键决策（非实现）
- 所有用户可见结论必须可追溯到 `version_id` 与证据链（如适用）。
- 门禁失败必须阻断晋级，并产出可行动的整改项清单。
- 可观测与审计字段是契约的一部分，必须被一致执行。

## 数据完整性语义（规格层）

### 引用可解析性（MUST）
- 任何引用到 `engine_id` 的字段（如 `depends_on[*]`、弃用/替代关系、审计记录等）在进入下游阶段前 MUST 可解析到注册表中的唯一记录。
- 引用不可解析 MUST 失败并阻断晋级/发布；同时 MUST 产出可审计的失败原因（见契约 `rejection_reasons`）。

### 幂等 / 去重（MUST）
- 系统 MUST 定义确定性的去重键，以避免重复写入造成的审计噪音与状态漂移；去重键仅允许由契约字段构成（禁止隐式外部状态）。
- 约定的最小去重键（规格级语义）：
  - 注册表记录：`engine_id`
  - 生命周期事件：`event_id`
  - 弃用/替代关系：`deprecated_engine_id` + `effective_version_id`
  - 审计记录：`audit_id`
- 对相同去重键的重复提交：
  - 若提交内容等价（在本 spec 定义的等价范围内）SHALL 视为幂等成功。
  - 若提交内容不等价 MUST 视为冲突并拒绝（例如 `ENGINE_ID_ALREADY_REGISTERED` / `VALIDATION_FAILED`）。

### 冲突处理（MUST）
- 冲突的判定与拒绝原因 MUST 可复现、确定性（相同输入得到相同拒绝代码）。
- 冲突 MUST 记录审计事件，且审计记录必须遵循最小化/脱敏原则。

## 运行时确定性与偏差口径（规格层）

### 确定性边界（SHALL）
- 在相同输入（含 `engine_id`/`version_id` 等契约字段）与相同治理视图（注册表与关系在同一版本语义下）重复执行时：
  - 决策输出（ALLOW/DENY/BLOCK、拒绝原因代码）SHALL 等价。
  - 关键产物写入（注册表条目、事件、映射、审计记录）在本 spec 定义的等价范围内 SHALL 等价。

### 等价范围（最小）
- 允许的非确定性字段（若存在）MUST 被显式标注并可审计，例如：
  - `created_at/updated_at/occurred_at` 时间戳
  - 系统生成的 `audit_id/event_id`（若未由调用方提供）
- 除上述显式允许项外，输出差异 MUST 视为偏差。

### 偏差检测与解释（MUST）
- 系统 MUST 具备偏差检测能力：当相同输入在相同版本语义下产生非等价输出时，必须形成可审计的偏差事件。
- 偏差事件 MUST 至少包含：`engine_id`、`version_id`、偏差类型、拒绝原因（若适用）、关联 `request_id/trace_id`（若适用）。

## 失败处理策略与信号产物（规格层）

### 失败分类（最小）
- **拒绝（DENY）**：输入缺失/非法、引用不可解析、冲突、越权等（通常不可重试）。
- **阻断（BLOCK）**：门禁检查失败、关键依赖不可用导致无法保证一致性/审计（通常需要人工整改或回滚）。
- **内部错误（INTERNAL_ERROR）**：非预期异常（可能可重试）。

### 策略（MUST/SHALL）
- 对于拒绝与冲突，系统 MUST 返回确定性的拒绝原因代码（见契约 `rejection_reasons`）并写入审计记录。
- 对于内部错误，系统 SHOULD 标记 `retryable=true` 并允许上游按策略重试；重复失败 MUST 触发阻断信号。
- 对于阻断场景，系统 MUST 形成可行动的整改项线索（最小：失败类别、影响面、关联标识符）。

### 信号产物（最小）
- metrics：失败计数、拒绝原因分布、阻断事件计数（见 `observability.md`）。
- logs/traces：可按 `engine_id`/`version_id`/`request_id`（若适用）定位。
- audit：对关键拒绝/阻断必须生成审计记录（见 `audit_log.md`）。

## 发布策略（Rollout）与回滚触发（规格层）

### 分阶段 rollout（最小）
- **Shadow**：在不影响对外行为的前提下并行生成治理决策与审计记录，用于验证一致性与拒绝原因分布。
- **Canary**：仅对一小部分治理写操作生效（例如按 `engine_id` 白名单或内部调用方范围），监控阻断/冲突率。
- **Gradual**：逐步扩大生效范围，直至全量；扩容步骤 MUST 可审计（记录 rollout 阶段与范围）。

### 回滚触发条件（MUST）
- 完整性校验失败率超过阈值（例如 `ls_engine_governance_registry_integrity_failures_total` 激增）MUST 触发回滚或阻断。
- 阻断事件持续上升且无法在短期内修复 MUST 触发回滚或降级到只读模式（若允许）。
- 审计记录不可用/不可导出 MUST 视为 Gate-0 阻断条件。

## Top 风险与缓解（写入门禁/证据/回退）

1) **注册表漂移/冲突**：重复注册或不一致更新导致状态不可追溯  
   - 缓解：幂等/冲突语义（本文件）+ 完整性校验（`data_dictionary.md`）+ 冲突审计事件（`audit_log.md`）

2) **引用不可解析导致下游错误**：`depends_on`/替代关系失效  
   - 缓解：引用可解析性阻断（本文件）+ 校验失败阻断策略（`acceptance.md`）

3) **高基数可观测导致成本失控**：将 `engine_id` 等作为 metric 标签  
   - 缓解：标签约束（`observability.md`）+ 成本阈值与告警（`cost.md`）

4) **越权/隔离失效**：非授权主体修改全局治理信息  
   - 缓解：访问控制/隔离边界（`security.md`）+ 审计最小字段与脱敏（`audit_log.md`）

5) **审计缺失导致不可追责**：关键拒绝/阻断未形成审计链路  
   - 缓解：审计字段契约（`audit_log.md`）+ Gate-0 失败阻断策略（`acceptance.md`）

## 非目标
- 不规定具体存储/消息队列/监控方案。
- 不包含任何代码路径、伪代码或接口实现。
