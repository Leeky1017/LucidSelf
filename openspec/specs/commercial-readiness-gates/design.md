# 商业化就绪门禁 Design

## 设计意图
本设计文档用于固定「商业化就绪门禁」在规格层面的设计立场：契约、数据完整性、门禁与审计。禁止写实现细节。

## 依赖关系
- 上游依赖：`deployment-release`, `security-privacy-compliance`, `billing-and-entitlements`

## 必备标识符（最小集）
- `version_id`

## 关键决策（非实现）
- 所有用户可见结论必须可追溯到 `version_id` 与证据链（如适用）。
- 门禁失败必须阻断晋级，并产出可行动的整改项清单。
- 可观测与审计字段是契约的一部分，必须被一致执行。

## 数据完整性语义（规格层）

### 引用可解析性（MUST）

- 发布许可证据中若包含证据链字段（例如 `chain_id` 或等价字段），其值 MUST 可解析到 `citations-evidence-protocol` 的 `EvidenceChain.chain_id`；不可解析 MUST 阻断（`CITATION_NOT_RESOLVABLE` 或 `VALIDATION_FAILED`）。

### 幂等 / 去重（MUST）

- 系统 MUST 定义确定性的去重键，以避免重复就绪清单、重复审批记录、重复支持预案或重复许可证据导致审计链路漂移；去重键仅允许由契约字段构成（禁止隐式外部状态）。
- 约定的最小去重键（规格级语义）：
  - 就绪清单：`checklist_id`
  - 审批记录：`approval_id`
  - 支持与应急预案：`plan_id`
  - 发布许可证据：`evidence_id`
- 对相同去重键的重复提交：
  - 若提交内容等价（在本 spec 定义的等价范围内）SHALL 视为幂等成功。
  - 若提交内容不等价 MUST 视为冲突并拒绝（`VALIDATION_FAILED`）。

### 冲突处理（MUST）

- 冲突的判定与拒绝原因 MUST 可复现、确定性（相同输入得到相同拒绝代码）。
- 冲突 MUST 记录审计事件，且审计记录必须遵循最小化/脱敏原则。

## 运行时确定性与偏差口径（规格层）

### 确定性边界（SHALL）

- 在相同输入（含 `version_id` 与就绪清单/审批链的同一快照）下重复评估时：
  - 输出 MUST 始终满足契约 schema 约束（结构可判定）。
  - 门禁结论（PASS/BLOCK）与阻断原因集合 SHALL 在本 spec 定义的等价范围内一致。

### 等价范围（最小）

- 允许的非确定性字段/差异 MUST 被显式标注并可审计：
  - `created_at` 等时间戳
- 除上述显式允许项外，门禁结论/阻断原因差异 MUST 视为偏差。

### 偏差检测与解释（MUST）

- 系统 MUST 能检测门禁评估偏差（例如同一 `version_id` 在同一输入快照下产生不同结论）。
- 偏差事件 MUST 至少包含：`version_id`、偏差类型与可行动摘要。

## 失败处理策略与信号产物（规格层）

### 失败分类（最小）

- **拒绝（DENY）**：标识符缺失/非法、越权等（通常不可重试）。
- **阻断（BLOCK）**：就绪清单不通过、审批链不完整、许可证据缺失（通常需要整改或补齐审批）。
- **内部错误（INTERNAL_ERROR）**：非预期异常（可能可重试）。

### 策略（MUST/SHALL）

- 对于拒绝与冲突，系统 MUST 返回确定性的拒绝原因代码（见契约 `rejection_reasons`）并写入审计记录。
- 对于就绪清单/审批链/许可证据缺失或不一致，系统 MUST 阻断商业发布并产出可行动整改线索（最小：失败原因 + 关联标识符 + 证据路径）。
- 对于内部错误，系统 SHOULD 标记 `retryable=true` 并允许上游按策略重试；重复失败 MUST 触发阻断信号。

### 信号产物（最小）

- metrics：门禁评估计数、阻断原因分布、审批链缺失计数、审计记录写入计数（见 `observability.md`）。
- logs/traces：可按 `version_id/request_id`（若适用）定位。
- audit：对关键拒绝/阻断/审批决策必须生成审计记录（见 `audit_log.md`）。

## 发布策略（Rollout）与回滚触发（规格层）

### 分阶段 rollout（最小）

- **Shadow**：不影响对外发布的前提下并行运行门禁评估与证据校验，仅记录差异与审计，用于校准清单项与阻断口径。
- **Canary**：仅对内部版本/小范围版本生效，监控阻断率、审批链完整性与评估延迟。
- **Gradual**：逐步扩大生效范围直至全量；扩容步骤 MUST 可审计（记录 rollout 阶段与范围）。

### 回滚触发条件（MUST）

- 阻断率异常上升（非预期的大面积 BLOCK）MUST 触发回滚或切换为 shadow 模式。
- 审计记录不可用/不可导出 MUST 视为 Gate-3 阻断条件。

## Top 风险与缓解（写入门禁/证据/回退）

1) **审批链不可追责**：缺失审批记录或审批人身份不可审计  
   - 缓解：审批记录契约（`contract/*.json`/`data_dictionary.md`）+ 审计导出（`audit_log.md`）+ Gate-3 阻断策略（`acceptance.md`）

2) **门禁被绕过**：发布路径未执行门禁或未写入许可证据  
   - 缓解：许可证据契约（`data_dictionary.md`）+ 回滚触发条件（本文件）+ Gate-3 阻断策略（`acceptance.md`）

3) **敏感信息泄露**：审批原因/证据中包含客户或合同敏感信息  
   - 缓解：最小化/脱敏（`security.md`/`audit_log.md`）+ 标签约束（`observability.md`）
