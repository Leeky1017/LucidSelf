# 部署与发布 — 安全与隔离边界（规格层）

本文件定义 `deployment-release` 在安全/访问控制/隔离方面的最小契约。禁止写实现细节；仅定义 MUST/SHALL 与可判定口径。

## 1) 访问控制（最小）

- 环境晋级、发布验证与回滚属于高敏感操作：
  - 非授权请求 MUST 被拒绝（`UNAUTHORIZED`）并写入审计记录。
  - 授权策略 MUST 可审计（审计记录至少包含 `actor_type`/`actor_id`（脱敏）与 `result`）。
- 发布产物清单、验证报告与回滚记录对外暴露 SHOULD 默认最小化：仅暴露必要字段；超出范围的字段 MUST 受权限控制。

## 2) 隔离边界（user_id/org_id 的适用方式）

- 本能力通常为“平台/运维域”能力；若发布/环境晋级存在租户隔离需求：
  - `org_id` MUST 被纳入授权范围与审计上下文，且不得跨越授权范围读取/写入敏感发布信息。
  - 若检测到跨边界访问意图，MUST 拒绝并写入审计记录（`UNAUTHORIZED`）。
- 若本能力不以 `user_id/org_id` 作为隔离边界，系统 MUST 明确其为“全局域”并定义可审计的授权范围；普通用户上下文不得越权调用。

## 3) 敏感信息最小化

- 本能力产生的 logs/metrics/audit MUST 不包含不必要敏感信息（密钥、token、部署凭据、访问密钥、环境秘钥等）。
- `actor_id` 等潜在敏感标识 SHOULD 采用脱敏或不可逆摘要；但 MUST 保持可追责关联性。
